## Objective
Create test files for the 4 private helper functions. These functions are not exported and cannot be called directly; they must be invoked using Pester's `InModuleScope` feature, which allows tests to execute code within the module's namespace where private functions are accessible.

## Pattern for Private Helper Tests
Each private helper test file follows this pattern:

```powershell
BeforeAll {
    . $PSScriptRoot/../Helpers/TestBootstrap.ps1
    Initialize-MDEValidatorTest
    
    . $PSScriptRoot/../Helpers/MockBuilders.ps1
}

Describe '<HelperName>' {
    Context 'Behavior scenario 1' {
        It 'does something when condition is true' {
            InModuleScope MDEValidator {
                # Call the private helper directly (only works inside InModuleScope)
                $result = <HelperName> -Parameter value
                
                # Assert result
                $result | Should -Should -Be expected_value
            }
        }
    }
    
    Context 'Behavior scenario 2' {
        It 'does something when condition is false' {
            InModuleScope MDEValidator {
                # Mock external calls that the helper makes
                Mock Get-ItemProperty { ... }
                
                $result = <HelperName>
                
                $result | Should -Be expected_value
                Should -Invoke Get-ItemProperty -Times 1
            }
        }
    }
}
```

## Specific Implementation for Each Helper

### 1. ConvertTo-HtmlEncodedString.Tests.ps1
This helper encodes special HTML characters. Test:
- Pass: Encodes &, <, >, ", ' correctly
- Fail: Handles empty/null input gracefully

Example test case:
```powershell
InModuleScope MDEValidator {
    $result = ConvertTo-HtmlEncodedString -InputString '<script>alert("xss")</script>'
    $result | Should -Match '&lt;script&gt;'
}
```

### 2. Write-ValidationResult.Tests.ps1
This helper creates standardized validation result objects. Test:
- Pass: Returns PSCustomObject with TestName, Status, Message, Recommendation properties
- Fail: Handles edge cases (null TestName, invalid Status values)

Example test case:
```powershell
InModuleScope MDEValidator {
    $result = Write-ValidationResult -TestName 'TestCase' -Status 'Pass' -Message 'Test passed'
    $result.TestName | Should -Be 'TestCase'
    $result.Status | Should -Be 'Pass'
}
```

### 3. Test-IsElevated.Tests.ps1
This helper checks for admin privileges by calling external APIs. Test:
- Pass: Mock to return true when admin
- Fail: Mock to return false when non-admin

This helper likely calls Get-ItemProperty or similar; use mocking:
```powershell
InModuleScope MDEValidator {
    Mock Get-ItemProperty { [pscustomobject]@{InstallationType='Client'} }
    $result = Test-IsElevated
    # Assert whether elevated or not
}
```

### 4. Test-IsWindowsServer.Tests.ps1
This helper checks if the operating system is Windows Server. Test:
- Pass: Mock to return true when Windows Server
- Fail: Mock to return false when Windows Client

Example:
```powershell
InModuleScope MDEValidator {
    Mock Get-ItemProperty { [pscustomobject]@{InstallationType='Server'} }
    $result = Test-IsWindowsServer
    $result | Should -BeTrue
}
```

## Implementation Notes
- Each test file must call Initialize-MDEValidatorTest in BeforeAll (imports module)
- InModuleScope allows private function invocation and module-scoped mocks
- The mock statements inside InModuleScope affect the module's namespace, not the test namespace
- Each helper should have at least one pass and one fail/edge-case context
- Use `Should -Invoke` inside InModuleScope to verify external calls if the helper calls commands like Get-ItemProperty
