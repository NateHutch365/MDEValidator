<decisions_made>

**Phase 2 Planning Decisions:**

1. **Test Organization:**
   - File-per-function layout: Tests/Public/<FunctionName>.Tests.ps1 and Tests/Private/<HelperName>.Tests.ps1
   - Separate directories for public functions and private helpers
   - Shared helper modules in Tests/Helpers (TestBootstrap.ps1 for module import, MockBuilders.ps1 for mock builders)

2. **Mocking Strategy:**
   - Use Mock -ModuleName MDEValidator for all external cmdlet dependencies
   - Mock 4 external cmdlets: Get-MpPreference, Get-MpComputerStatus, Get-Service, Get-ItemProperty
   - Private helpers use InModuleScope for invocation and module-scoped mocking (TEST-06 requirement)

3. **Test Execution:**
   - Migrate from direct Invoke-Pester to New-PesterConfiguration approach (Pester 5.x pattern)
   - JaCoCo coverage output to Tests/Artifacts/coverage.xml (TEST-05 requirement)
   - Tests must run without admin privileges and without Defender installed (TEST-04 requirement)

4. **Verification Approach:**
   - Parity validation: Compare new split test suite outcomes against old monolithic suite before cutover
   - Mapping inventory: Generate function-test-map.json showing test coverage status (45/45 public + 4/4 private)
   - Automated verification scripts for each task checkpoint

5. **Implementation Sequence:**
   - Wave 1: Infrastructure first (folders, helpers, runner configuration)
   - Wave 2: Public function tests (45 functions, grouped by dependency families)
   - Wave 3: Private helper tests (4 helpers with InModuleScope)
   - Sequential execution: Each plan depends on previous plan completion

**Rationale:**
- File-per-function keeps tests modular and parallel-safe
- Mock-based approach eliminates Defender/admin dependencies and enables deterministic testing
- InModuleScope is required for testing non-exported private helpers
- JaCoCo output enables downstream CI/CD coverage reporting
- Parity validation ensures migration doesn't lose coverage
</decisions_made>

<context>

**Mental Model:**

Phase 2 is about making the test suite **portable and deterministic**. Currently, tests implicitly depend on:
- Defender being installed (Get-MpPreference, Get-MpComputerStatus)
- Running as administrator (some registry paths, service controls)
- Specific machine state (registry values, service status)

The new architecture removes these dependencies through mocking, making tests runnable in CI/CD pipelines and on developer machines without special setup. The trade-off is test complexity (more mock setup code), but we gain parallelization, speed, and reliability.

**The "Why":**
- TEST-01: Coverage completeness enables confidence in refactoring
- TEST-02: Mocking eliminates environment dependencies
- TEST-03: Pass/fail scenarios catch regressions in both directions
- TEST-04: No-admin/no-Defender enables CI/CD and developer workflows
- TEST-05: JaCoCo coverage enables quality gates and coverage trends
- TEST-06: Private helper coverage completes the test pyramid
